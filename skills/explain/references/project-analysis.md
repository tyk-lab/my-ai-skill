## 1. 分析目标

把项目解释成一套完整的软件工程系统，而不是目录清单、零散调用链或互不相干的图。

始终建立以下阅读顺序：

`核心认知 → 总体框架 → 构建与启动 → 端到端核心行为 → 数据与状态 → 时间/并发 → 资源生命周期 → 外部边界 → 故障恢复 → 验证与演进`

“完整”表示主线每一环都有可信答案，不表示穷举所有文件、函数或次要功能。优先选择最能解释系统目标的 1-3 条真实业务链、控制链或数据链，并说明其它模块如何挂接到主线。

## 2. 项目分型

先按证据选择分析档，再开始确定图集：

- `embedded`：直接控制 MCU、SoC、板卡或硬件，存在 HAL、寄存器、ISR、DMA、Pin、Timer、RTOS 任务、固件镜像或烧录流程。
- `general`：运行在 OS / 容器 / 浏览器 / 虚拟机之上，以进程、线程、请求、事件、任务、数据库、缓存或外部服务为主要运行边界。
- `docs`：主要交付物是文档、知识结构、公式推导、脚本、演示和配图，而非可执行软件。
- `mixed`：仓库同时包含多个运行模型。按子项目分别应用对应分析档，再生成一张跨子项目集成边界图；不要用一个折中模板覆盖全部。

自动判断只能作为初选。用构建入口、运行入口、目录职责和实际依赖校验；关键边界仍不清时再向用户确认。

## 3. 统一软件工程主线

每个代码型项目必须回答以下问题，并把答案串成连续因果链：

1. **核心认知**：为谁解决什么问题；核心价值、领域对象、成功标准、3-5 个理解要点和不能破坏的约束是什么。
2. **总体框架**：系统由哪些职责单元组成；依赖方向、入口/出口、控制边界、数据边界和平台边界是什么。
3. **构建与启动**：源码、配置、依赖和生成物如何形成可运行系统；启动、注册、加载和停止顺序是什么。
4. **端到端行为**：一个代表性输入如何穿过系统，改变状态、调用核心机制并形成输出或反馈闭环。
5. **数据与状态**：配置、运行态、持久化状态和派生数据由谁创建、校验、转换、保存及恢复。
6. **时间与并发**：有哪些执行上下文；谁调度谁；共享状态如何同步；超时、周期、优先级和竞态边界是什么。
7. **资源生命周期**：资源由谁分配或注册、持有、访问、停用或释放；如何发现冲突和处理失败。
8. **外部边界**：与用户、硬件、协议、API、数据库、文件、消息或第三方系统交换什么，契约和失败边界是什么。
9. **故障与恢复**：错误如何检测、传播、隔离、记录、回退和恢复；安全边界与不可逆动作如何保护。
10. **验证与演进**：怎样构建、测试、调试、观测和发布；修改某条主线会影响哪里；扩展点和主要技术风险是什么。

每一环至少回答：`为什么存在｜谁负责｜输入输出｜依赖状态｜运行机制｜资源/并发｜失败行为｜验证方式｜修改影响`。不适用项写明项目证据，不用通用知识猜测。

## 4. 代码型项目必出图

embedded 和 general 共同生成以下图；内容按所选分析档填充：

1. `00-核心认知与阅读地图.drawio`：项目目的、核心价值、领域对象、3-5 个核心要点、关键约束、图集阅读主线。
2. `01-软件工程总体框架.drawio`：静态职责、运行时单元、数据/状态、资源、平台与外部边界组成的整体框架；不是目录树。
3. `02-构建启动与运行生命周期.drawio`：从源码/配置到产物、启动、初始化、稳态运行、停止或重启。
4. `03-端到端核心主线.drawio`：1-3 条代表性输入到输出/反馈闭环，标出状态变化和关键交接点。
5. `04-核心行为与调用链.drawio`：主线上的真实函数、组件、任务或处理器调用；说明为何是核心路径，不画全量调用图。
6. `05-数据配置与状态.drawio`：配置来源、数据转换、运行状态、持久化、缓存及恢复关系。
7. `06-执行模型时间与并发.drawio`：执行上下文、调度、时间机制、共享状态、同步和竞态边界。
8. `07-资源所有权与生命周期.drawio`：关键资源的分配/注册、所有者、访问者、停用/释放、冲突检测和失败恢复。
9. `08-外部接口与集成边界.drawio`：用户、硬件、协议、API、文件、数据库、消息或第三方系统的输入输出契约。
10. `09-故障安全诊断与恢复.drawio`：错误检测、传播、日志/诊断、安全保护、降级、回退和恢复。
11. `10-构建测试调试与演进.drawio`：构建矩阵、测试层次、调试/观测入口、发布路径、扩展点和修改影响。

这些图承担不同主线职责，不能以“已并入其它图”为理由省略。若两个主题确实高度重合，可在用户明确同意后合并，但必须保留两个主题的完整问题集和文档章节。

## 5. 嵌入式 / 固件专项图

embedded 在共同图之外必须生成：

1. `11-中断DMA与实时执行.drawio`：中断源、触发方式、优先级/嵌套、屏蔽/临界区、DMA 通道、ISR 与主循环/任务的数据交接、注册和停用路径。
2. `12-MCU时钟时间与调度.drawio`：系统时钟树、Tick、硬件 Timer、软件超时、周期任务、单位和溢出处理、实时截止时间及优先级关系。
3. `13-内存布局与管理.drawio`：Flash/RAM 分区、静态区、栈、堆、固定缓冲区、DMA/缓存/对齐、所有权、容量边界和监测方式。
4. `14-外设资源所有权与冲突.drawio`：Pin mux、外设时钟、Timer/UART/SPI/I2C/ADC/PWM、IRQ/DMA 通道的映射、初始化、占用、复用、停用/释放和冲突检查。
5. `15-芯片板级与安全约束.drawio`：HAL、芯片/板卡差异、寄存器与电气/时序前提、看门狗、危险输出、编译期校验和硬件故障边界。

如果项目使用 RTOS，在 `06` 和 `11` 中补任务、调度器、队列、信号量、ISR-safe API 与优先级反转；如果是 bare-metal，明确主循环、ISR、轮询和协作式状态机之间的关系。

嵌入式资源必须逐项回答：`资源标识｜配置来源｜所有者｜初始化/注册｜运行期访问者｜ISR/DMA/并发保护｜停用/释放｜复用冲突｜失败恢复`。长期占用直到复位也属于明确的生命周期结论，不能因“没有 release”而省略。

## 6. 通用软件专项图

general 在共同图之外，根据实际运行模型生成以下必需专项图；不适用的相邻主题可以合并，但不能缺少问题答案：

1. `11-进程线程异步与任务生命周期.drawio`：进程/线程、请求、事件循环、异步任务、队列、调度、取消、超时和共享状态。
2. `12-存储事务缓存与一致性.drawio`：数据库、文件、缓存、消息、事务边界、一致性、幂等、版本与恢复。
3. `13-运行资源池与生命周期.drawio`：连接池、线程池、句柄、会话、缓存、临时文件和外部客户端的创建、租用、归还、关闭及耗尽处理。
4. `14-部署观测与运行安全.drawio`：配置/密钥、部署拓扑、健康检查、日志/指标/追踪、认证授权、扩缩容、滚动升级和故障隔离。

库/SDK 强调公共 API、对象所有权、兼容性和调用方责任；CLI 强调命令解析、进程退出码、文件/终端副作用；前端强调组件状态、路由、请求、缓存和浏览器生命周期；服务端强调请求、任务、存储、事务和运行观测。

## 7. 文档型仓库输出

docs 不硬套软件运行模型，生成：

1. `00-核心认知与阅读地图.drawio`
2. `01-知识框架与职责分层.drawio`
3. `02-推荐阅读与学习主线.drawio`
4. `03-主题推导演示引用链.drawio`
5. `04-内容生成验证与维护.drawio`
6. `05-读者入口与外部工具边界.drawio`

等价主线是：知识目标 → 内容框架 → 阅读路径 → 主题/推导 → 脚本/配图/演示 → 验证维护。公式推导只放在项目约定的推导层，不散落回主题层。

## 8. 总描述文档

所有图完成并核对后，最后生成 `.project-analysis/PROJECT_ANALYSIS.md`。这是分析正文和总描述，不是概览、文件索引或图注集合。

正文至少包含：

1. **分析范围与证据基线**：分析档、目标范围、已读入口、版本/日期或提交基线。
2. **核心认知**：项目本质、价值、核心对象、关键约束、3-5 个核心要点。
3. **软件工程总体框架**：把静态架构、运行架构、数据/状态、资源、平台和外部边界连成一体。
4. **完整工程主线**：从构建、启动、输入到核心行为、状态变化、资源使用、输出、失败恢复和验证演进，写成连续因果叙事。
5. **分析档专项机制**：embedded 或 general 的关键机制、约束和工程取舍。
6. **逐图说明**：按主线顺序嵌入每张已生成图；解释它回答什么、核心结论、关键节点/箭头、与前后图如何衔接、证据、风险和修改影响。
7. **验证入口与关键不确定项**：给出可执行验证入口，只列会影响结论的未知事实，不罗列“未出图原因”。

逐图说明禁止只写“该图展示了……”或复述文件名。每张图的文字必须包含真实模块/函数/资源/状态，并让没有打开 drawio 的读者仍能理解其主要结论。

禁止生成或保留 `PROJECT_OVERVIEW.md`。如果目标目录已有旧文件，覆盖/删除属于高风险文件操作，先征得用户确认；未获确认时只生成新文件并报告旧文件仍存在。

## 9. 取证与完整性门禁

取证顺序：项目索引/上下文 → 构建与配置入口 → 启动/运行入口 → 主线模块 → 测试/调试入口 → 专项机制。已有上下文只能作为导航，必须用当前源码、配置和测试做新鲜度校验。

为每个主线环节维护内部覆盖表：

`主线环节｜核心结论｜代码/配置/测试证据｜对应图｜文档章节｜状态`

交付前确认：

- 所有共同必出图均已完成，所选分析档的专项必出图均已完成。
- 每个核心结论至少有一个直接证据；关键跨模块结论通常需要两个以上相互印证的锚点。
- 图中的模块、函数、状态、资源和路径存在，行号与当前版本一致。
- 不用 `[待补充]`、`[需深挖]`、空节点或通用模板文字冒充完成；真正无法确认的事实集中标 `[未确认]` 并说明影响。
- 不把资源、错误、并发、初始化等横切关注点拆散后宣称已经完整覆盖；总描述必须重新串回主线。

## 10. 图与文档质量门禁

每张图必须有：明确问题、中心结论、真实节点、带语义的箭头、边界/约束、`file:line` 证据，以及它在工程主线中的位置。避免目录树、无方向方框堆叠、纯清单和全量调用图。

`PROJECT_ANALYSIS.md` 必须通过以下检查：

- 开头能让读者在一分钟内说出项目是什么、核心价值和最重要的理解主线。
- 总体框架同时说明静态职责和动态协作，不只列目录。
- 至少一条端到端路径贯穿输入、处理、状态、资源、输出和失败行为。
- 每张生成图都在正文中得到实质解释，并与前后章节建立关系。
- 嵌入式项目完整覆盖 ISR/DMA、MCU 时间、内存、外设所有权和板级安全；通用软件完整覆盖执行上下文、数据一致性、运行资源和部署观测。
- 结尾给出验证与修改入口，而不是泛泛总结或未生成文件清单。
